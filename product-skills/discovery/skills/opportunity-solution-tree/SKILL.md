# Opportunity Solution Tree (Teresa Torres)

## Purpose
Visualize the connection from desired outcome → opportunities → solutions → experiments to avoid solution bias and ensure discovery is outcome-driven.

## OST Structure

```
DESIRED OUTCOME
      |
  ----+----+----+----
  |        |        |
OPPORTUNITY OPPORTUNITY OPPORTUNITY
  |        |        |
 SOL     SOL SOL   SOL SOL
  |        |    |    |   |
 EXP     EXP  EXP  EXP EXP
```

## How to Build the OST

### Level 1: Desired Outcome
- Must be a measurable business metric the team can influence
- Not a solution ("launch feature X") — an outcome ("increase 30-day retention by 10%")
- One outcome per tree

### Level 2: Opportunity Space
Opportunities = unmet needs, pain points, desires of your customers
- Gather from: interviews, support tickets, NPS comments, session recordings
- Express as customer needs: "Users struggle to understand what to do next after signup"
- NOT as solutions: "Add tooltips" is NOT an opportunity
- Cluster related opportunities into themes
- Dig deeper: each opportunity can have sub-opportunities

### Level 3: Solution Space
For each opportunity, generate multiple solutions:
- At least 3 solutions per opportunity (avoid locking into one)
- Solutions can be features, process changes, content, pricing changes, etc.
- Ask: "What else could solve this opportunity?"

### Level 4: Experiment Space
For each solution, design the smallest experiment to test the riskiest assumption:
- Fake door test
- Prototype
- Concierge test
- Landing page test
- Wizard of Oz

## Output Format

Present as a structured tree with:
- Outcome (1)
- Opportunities (3-5 top-level, with sub-opportunities)
- Solutions (2-4 per opportunity)
- Experiments (1-2 per solution)

Include a "Where to Focus" recommendation based on opportunity sizing and risk.
