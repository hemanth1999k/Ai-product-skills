# Structured Problem Solving

## Purpose
Apply rigorous, structured thinking to any product or business problem — from metric drops to strategic challenges to operational failures.

## Problem Solving Framework (5D Method)

### D1: Define
Before solving, define precisely:
- What IS the problem? (facts, data, symptoms)
- What is NOT the problem? (scope the boundary)
- Who is affected? (user segment, team, stakeholder)
- When did it start? (timeline)
- How severe? (impact quantified)

A problem well-defined is half-solved.

### D2: Diagnose
Find the root cause, not the symptom:
- Use the 5 Whys (ask "why" 5 times)
- Use fishbone / Ishikawa diagram (People, Process, Technology, Environment, Measurement, Materials)
- Use the TROPIC framework for metric problems (see rca-tropic skill)
- Gather data: What does the data say about where in the funnel/flow the break is?

### D3: Diverge
Generate multiple possible solutions:
- Minimum 3 directions before evaluating any
- Include quick wins AND systemic fixes
- Include "do nothing" as an option (what happens if we wait?)
- Ask: Who has solved this before? What can we steal?

### D4: Decide
Evaluate options on:
- Impact (how much does it fix the problem?)
- Effort (how long / how much resource?)
- Risk (what could go wrong?)
- Reversibility (can we undo this?)

Use a decision matrix. Don't just pick the obvious answer.

### D5: Deploy
- Define the smallest test of the solution
- Set success/failure criteria BEFORE deploying
- Define rollback plan
- Assign owner and timeline

## Common Problem Patterns & Solutions

| Problem Pattern | Likely Cause | Approach |
|---|---|---|
| Metric dropped suddenly | External event, bug, data issue | Timeline analysis, isolate segment |
| Metric declining slowly | Product-market fit erosion, competition | Cohort analysis, user interviews |
| Team repeatedly stuck | Process/communication issue | Retrospective, process redesign |
| Feature not adopted | Awareness, usability, or value gap | Funnel analysis, usability test |
| High churn | Onboarding failure or value not delivered | Cohort analysis, exit interviews |

## Output Format
- Problem statement (refined)
- Root cause analysis
- Top 3 solution options with trade-offs
- Recommended next step
